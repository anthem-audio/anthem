# Change Events and Listeners

## Scope

This page documents the change-event system for synced project models:

- the shape of change events in Dart (`ModelChangeEvent`)
- where those events are created
- how to listen to events in Dart (`onChange(...)`, raw listeners, filter builders)
- how change events are converted into engine update messages

Deep C++ update application details are covered in the engine-side article.

## Event Shape in Dart

Each model change event is represented by `ModelChangeEvent`, which has three payload fields:

1. `fieldAccessors`: path from root model to the changed location
2. `operation`: what changed at that location
3. `sendToEngine`: whether the root model should forward the change to engine IPC

The event objects come from `AnthemModelBase`:

- `ModelChangeEvent`
- `FieldAccessor`
- `FieldOperation` and subclasses

### `FieldAccessor` path structure

A `FieldAccessor` describes one hop in the model path.

Fields:

- `fieldType`: `raw`, `list`, or `map`
- `fieldName`: set for raw field access
- `index`: set for list element access
- `key`: set for map value access

For listener callbacks, accessors are provided in **root-to-leaf** order.

Examples:

1. `project.name = 'X'`
   - `[ FieldAccessor(raw, fieldName: 'name') ]`
2. `project.subElement.value = 'Y'`
   - `[ FieldAccessor(raw, 'subElement'), FieldAccessor(raw, 'value') ]`
3. `project.listOfSubElements[0].value = 'Y'`
   - `[ FieldAccessor(raw, 'listOfSubElements'), FieldAccessor(list, index: 0), FieldAccessor(raw, 'value') ]`
4. `project.mapOfSubElements['one'].value = 'Y'`
   - `[ FieldAccessor(raw, 'mapOfSubElements'), FieldAccessor(map, key: 'one'), FieldAccessor(raw, 'value') ]`

### `FieldOperation` types

The operation encodes the mutation kind and carries typed payloads.

Supported operations:

- `RawFieldUpdate`
  - for direct field assignment (`myField = ...`)
  - includes `oldValue`, `newValue`, `newValueSerialized`
- `ListInsert`
  - for list insert/add
  - includes `value`, `valueSerialized`
- `ListUpdate`
  - for list index replacement
  - includes `oldValue`, `newValue`, `newValueSerialized`
- `ListRemove`
  - for list remove
  - includes `removedValue`
- `MapPut`
  - for map insert/update
  - includes `oldValue`, `newValue`, `newValueSerialized`
- `MapRemove`
  - for map remove
  - includes `removedValue`

For each operation, both runtime values and serialized values are available where needed. Serialized values are what the model-sync pipeline can forward directly to engine messages.

### Dart-only events

Fields marked with `@hideFromCpp` still generate normal Dart change
events so `onChange(...)` and raw listeners can observe them. Those events set
`sendToEngine` to `false`, which tells the root `ProjectModel` listener to skip
IPC for that change.

## Where Events Are Created

### Generated synced setters (raw fields)

In generated Dart mixins, sync-enabled field setters:

1. capture old value
2. assign new value
3. attach/detach child model metadata when applicable
4. call `notifyFieldChanged(...)` with a `RawFieldUpdate` plus the local accessor chain

This is generated by `AnthemModelGenerator` for fields participating in change tracking.

### `AnthemObservableList` and `AnthemObservableMap`

Collection wrappers observe MobX collection mutations and emit list/map operations.

Examples:

- list add/insert -> `ListInsert`
- list index assignment -> `ListUpdate`
- list remove -> `ListRemove`
- map put/update -> `MapPut`
- map remove -> `MapRemove`

They also handle parent metadata updates and child detach behavior so future events remain valid.

## Dart Listener APIs

### Low-level listeners

`AnthemModelBase` provides raw subscription methods:

- `addRawFieldChangedListener((change) { ... })`
- `removeRawFieldChangedListener(...)`

Use this when you need full unfiltered event access.

### Generated `onChange(...)`

Synced model classes get a generated API:

```dart
final sub = model.onChange(
  (b) => b.notes.anyValue.anyElement.anyField,
  (event) {
    // event.fieldAccessors, event.operation
  },
);
```

`onChange(...)` takes:

1. a filter-builder function
2. a listener callback receiving `ModelChangeEvent`

It returns `ModelFilterSubscription` with `cancel()`.

### Filter-builder concepts

The generated builder DSL composes path and behavior filters.

Common building blocks:

- field getters (for example `b.name`, `b.loopPoints`)
- `anyField`
- list `anyElement`
- map `anyValue`
- `multiple([...])` (logical OR of sub-filters)
- `filterByChangeType([...])`
- `withDescendants`

`withDescendants` allows matching deeper changes below the currently selected node. Without it, a filter usually matches only the exact level.

Fields marked `@hideFromCpp` remain available in this builder DSL. They are
hidden only from C++ generation and engine sync, not from Dart listeners.

### Change-type filtering

`filterByChangeType([...])` supports:

- `fieldUpdate`
- `listInsert`
- `listRemove`
- `listUpdate`
- `mapPut`
- `mapRemove`

This is frequently used to distinguish structural collection mutations from descendant field mutations.

### Subscription lifecycle

For long-lived model-level listeners, keeping subscriptions for model lifetime is typical.  
For temporary listeners (for example, scoped logic), call `cancel()` when done.

## Conversion to Engine Sync Messages

At the project root, Anthem bridges raw model events to engine requests in `ProjectModel._attachModelChangeListener()`. This conversion does not change the meaning of the events, but encodes them in the form of IPC objects from [`/lib/engine_api/messages/model_sync.dart`](/lib/engine_api/messages/model_sync.dart).

Conversion steps:

1. skip the event entirely if `sendToEngine == false`
2. map each `FieldAccessor` to message `FieldAccess` (`fieldType`, `fieldName`, `listIndex`, `serializedMapKey`)
3. map `FieldOperation` to `FieldUpdateKind`
   - `RawFieldUpdate`, `ListUpdate`, `MapPut` -> `set`
   - `ListInsert` -> `add`
   - `ListRemove`, `MapRemove` -> `remove`
4. select serialized payload when needed
   - `RawFieldUpdate.newValueSerialized`
   - `ListInsert.valueSerialized`
   - `ListUpdate.newValueSerialized`
   - `MapPut.newValueSerialized`
5. send `ModelUpdateRequest` via `engine.modelSyncApi.updateModel(...)`

Map keys are serialized to strings for transport (`serializedMapKey`) and deserialized on the C++ side according to the expected key type.

## C++ Listener Surface (Brief)

The generated C++ wrapper classes can expose per-field observer hooks when wrapper generation + model sync are enabled:

- `add<FieldName>Observer(...)`
- `remove<FieldName>Observer(...)`

Generated C++ update handlers call these field observer collections after applicable updates.  
There is also model-level observer support in `AnthemModelBase` on the engine side (`addObserver`, `processChange`).

The detailed C++ behavior and constraints are covered in the engine-side model-sync article.
